```
Problem: Design a rate limiter for 50M users, 10M req/sec

My process:
  1. Clarify constraints first (always). SLOs? Exactness vs speed tradeoff?
  2. Token bucket > leaky bucket for burst tolerance at this scale
  3. Avoid centralized state — sliding window w/ Redis cluster sharding
     by (user_id % N) across 64 shards → O(1) per check, ~40µs p99
  4. Local cache layer (in-process) handles 95% of traffic without
     network hop → Redis only for cross-instance coordination
  5. Fallback: fail open under network partition (Netflix-style)
     and alert, never degrade user experience silently
  6. Observability first: histogram of limiter decisions, shard hot-spots,
     eviction rates → alerts before users notice

Code I'd write first: the rejection path. Make the unhappy path fast.
Code I'd write last: the config layer. Premature flexibility is debt.

Rule I live by: measure before optimizing. intuition is a hypothesis.
```
